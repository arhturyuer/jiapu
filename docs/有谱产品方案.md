# 「有谱」产品方案

> 版本：V1.0 产品重构方案
> 产品形态：微信小程序 + 微信云开发运营管理后台
> 核心主张：一家人，共修一份家谱。

## 1. 产品定义

### 1.1 产品定位

「有谱」是一款面向普通家庭的协作式家谱小程序。用户不需要理解专业谱牒知识，只需选定一个起始成员，通过点击关系节点添加父母、伴侣、子女等成员，即可快速建立家谱；随后将家谱卡片分享给家人或家庭群，邀请家人直接浏览完整谱系，或从任意成员视角理解关系、补充资料、共同维护。

产品不是一次性的家谱制作工具，而是一个可以长期使用的“家庭数字空间”。家谱是核心骨架，家庭成员共同维护、资料更新、生日纪念和家族故事沉淀构成长线价值。

核心协作模型固定为：**一个人快速创建，一家人共同补全，由少数管理员维护秩序。** 首次建谱不依赖其他家人加入；协作发生在家谱已经初步成形之后。

### 1.2 核心价值

- 创建简单：由一个人先建起来，像搭积木一样添加亲属，不从复杂表单开始。
- 家人共建：一键生成微信分享卡片，家人打开后可以直接看全谱、加入和补资料。
- 关系易懂：既支持不带人物视角的完整谱系浏览，也可以随时选择任意成员为中心理解全家关系。
- 关系直观：用可缩放、可点击的关系图展示家谱，列表仅作为辅助入口。
- 长期可用：家谱保存在云端，可持续补充、纠错、交接管理员和导出备份。
- 私密可信：默认仅家庭成员可见，敏感资料分级展示，所有修改可追溯。

### 1.3 产品目标与非目标

第一阶段目标：

1. 让新用户在 3 分钟内完成建谱，并至少录入 3 位家庭成员。
2. 让创建者能够把家谱准确分享给指定家人或家庭群。
3. 让受邀家人无需认领身份，即可完成“加入—选择视角—理解关系—补充资料”。
4. 支持多人长期维护，并能解决误改、重复成员和关系错误。
5. 为运营人员提供合规、可审计的用户与内容治理后台。

第一阶段不做：

- 宗亲会、族谱编修委员会等重型组织管理。
- 面向陌生人的公开家谱广场。
- 自动从通讯录或微信群读取家庭关系。
- 复杂的古籍 OCR、宗谱排版和印刷出版。
- 强社交信息流、群聊或替代微信家庭群的即时通信。

## 2. 用户与使用场景

### 2.1 目标用户

| 用户 | 典型需求 | 设计重点 |
|---|---|---|
| 发起人 | 快速建立家谱并邀请全家参与 | 建谱引导、批量分享、管理权限 |
| 家庭管理员 | 整理成员、审核修改、处理重复关系 | 待办中心、关系校验、版本记录 |
| 普通家人 | 直接浏览完整家谱，或从熟悉的成员视角理解关系、补充资料 | 低门槛加入、双模式查看、清晰权限 |
| 年长家人 | 查看亲属关系、纠正信息、讲述家族故事 | 大字号、大按钮、少步骤、语义清楚 |
| 浏览成员 | 查看已被分享的家谱，但暂不参与编辑 | 明确隐私说明、轻量授权、引导加入 |
| 运营人员 | 处理反馈、违规和系统异常 | 最小化数据访问、审计、工单化处理 |

### 2.2 核心场景

1. 家庭聚会时，发起人现场建立家谱，边问边录入。
2. 发起人把家谱邀请卡片发到家庭群，群成员陆续加入。
3. 家人选择自己熟悉的成员作为查看视角，理解关系并补头像、生日、人物简介等资料。
4. 家庭管理员收到修改申请，审核关系变更和成员合并。
5. 平时从群内卡片、最近使用或“我的小程序”返回查看家谱。
6. 成员生日、纪念日或资料待补时，在用户主动订阅后发送提醒。

## 3. 产品核心闭环

```mermaid
flowchart LR
    A["从我开始创建家谱"] --> B["图形化添加父母/伴侣/子女"]
    B --> C["生成家庭邀请卡片"]
    C --> D["分享给家人或家庭群"]
    D --> E["家人加入并选择查看视角"]
    E --> F["理解关系、补充资料与纠错"]
    F --> G["管理员审核/自动合并"]
    G --> H["家谱更完整，持续查看与维护"]
    H --> C
```

核心增长不是“拉新注册人数”，而是家谱完整度提升带来的家庭协作：每新增一位家人，都可能补全更多成员与关系，并带来下一轮分享。

## 4. 小程序端产品方案

### 4.1 信息架构

底部导航精简为三个一级入口：

| 导航 | 定位 | 主要内容 |
|---|---|---|
| 家谱 | 核心工作台 | 图谱、搜索、添加亲属、切换家谱 |
| 家庭 | 协作与动态 | 家庭成员、待办、最近修改、邀请家人 |
| 我的 | 个人与设置 | 我的家谱、最近查看、消息、隐私、导出 |

不再把“首页、家谱树、成员列表”并列为同等层级。打开小程序默认进入最近访问的家谱图，减少一次导航；成员列表被收纳为家谱页的搜索/列表视图。

### 4.2 首次进入

首屏只提供三个明确选择：

- 创建新家谱
- 接受家人邀请
- 我先看看示例

用户从分享卡片进入时，直接展示邀请落地页，不先展示通用首页。落地页说明“谁邀请你查看哪一家谱、将以哪位成员为初始视角、加入后能看到什么、你将拥有什么权限”，确认后进入家谱。

### 4.3 创建家谱流程

将现有“先填家谱名称”的表单流程改成关系引导，由一个人独立完成首次创建：

1. 添加第一个成员：可以是自己，也可以是最熟悉的长辈，填写姓名或称呼，可选头像。
2. 选择要添加的人：父亲、母亲、伴侣、儿子、女儿。
3. 点击关系卡片快速录入：默认只要求姓名/称呼，其他信息以后补。
4. 系统根据发起人的姓氏建议家谱名称，如“张氏家谱”，用户可修改。
5. 达到 3 位成员后展示首张家谱图，并突出“邀请家人一起补全”。

设计原则是“先成图、后补资料”。姓名之外的生日、籍贯、简介等全部为渐进式填写，避免首次建谱被长表单打断。

### 4.4 家谱主界面

家谱页采用无限画布式关系图，并提供“全谱”和“成员视角”两种查看方式。默认进入全谱模式，顶部保留少量高频操作：

- 左上：家谱名称，点击切换家谱。
- 顶部中间：查看方式切换——“全谱 / 成员视角”；进入成员视角后显示“从张建国看”。
- 右上：分享。
- 画布浮动按钮：全谱适配、缩放、切换横向/纵向布局。
- 底部主按钮：添加亲属。

全谱模式下，成员节点展示头像、姓名和必要的生卒信息，通过父母—子女线、伴侣线和代际层级直接表达整个家谱结构，不显示“与谁的关系”。点击节点展开人物卡，不直接跳转复杂详情页；卡片提供“以此人为视角、查看资料、添加亲属、编辑、从此人分享”五个操作。

空关系位可以显示淡化的“＋父亲”“＋伴侣”“＋子女”节点。用户点击空位后，系统已经知道关系类型，只需输入姓名，减少手动选择父母 ID 的操作。

### 4.5 全谱与任意成员视角

#### 全谱模式

全谱模式是不带任何人物立场的中性家谱视图，也是普通打开家谱时的默认模式。它回答的是“这个家谱由哪些人、哪些关系组成”。

- 自动把当前可见谱系缩放到合适范围。
- 以代际层级、父母—子女连接线和伴侣连接线展示结构。
- 支持拖动、缩放、分支折叠、姓名搜索和定位成员。
- 不显示“父亲、岳父、表姐”等相对称谓，避免用户误以为当前绑定了某个人。
- 搜索定位后仍保持全谱模式，除非用户主动选择“以此人为视角”。

#### 成员视角模式

“视角成员”是查看家谱时的临时中心点，不等于当前微信用户本人，也不构成身份认证或资料所有权。

用户可以通过三种方式切换视角：

- 点击任意成员卡片，选择“从他/她看家谱”。
- 点击顶部“当前视角”，按姓名搜索成员。
- 打开家人分享的成员视角卡片，直接以卡片指定成员为中心。

切换后，系统重新计算并展示该成员与其他人的关系。例如同一个“张国强”，从张建国视角看可能是“父亲”，从李晓明视角看可能是“岳父”，从张小雨视角看可能是“爷爷”。当前视角节点使用明显的中心光环或定位标记，页面顶部持续显示“正在从谁的视角查看”，避免用户迷失。

关系称谓只在关系路径明确时显示；无法可靠推导时展示“姻亲”“亲属”或关系路径，不强行给出可能错误的称谓。用户可以随时点击“返回全谱”退出人物视角。系统可以保存用户最近使用的查看模式和成员视角，但从普通家谱入口进入时优先展示全谱；只有成员视角分享卡才直接进入指定视角。

视角切换只影响家谱布局、称谓和分享入口，不改变任何数据权限。一个访客即使切换到管理员对应的人物节点，也仍然只有访客权限。

### 4.6 图形化添加亲属

添加关系采用“选中心人物 → 选关系 → 填最少信息”的三步交互：

1. 默认以当前选中成员为中心。
2. 使用环形或底部关系面板选择父亲、母亲、伴侣、儿子、女儿、兄弟姐妹。
3. 输入姓名，选择“新建成员”或“关联已有成员”。

系统自动推导可确定的反向关系，例如给甲添加父亲乙时，同时建立乙到甲的子女关系。对于兄弟姐妹等需要共同父母才能严谨表达的关系，系统引导确认共同父亲/母亲，不直接生成模糊关系。

关系建立前进行实时校验：

- 避免成员成为自己的祖先或后代。
- 避免同一成员重复添加。
- 提醒明显冲突的父母、伴侣关系。
- 对无法自动判断的复杂关系提交管理员确认。

### 4.7 成员资料

成员资料分三层展示：

1. 基础资料：姓名、称呼、性别、头像、在世状态。
2. 家庭资料：生日、籍贯、居住地、人物简介、照片。
3. 敏感资料：完整出生日期、联系方式等，默认仅管理员和被明确授权的家庭成员可见。

成员资料页增加：

- “从此人视角查看”入口。
- 资料完整度提示，如“还差生日和头像”。
- 修改记录和纠错入口。
- 家族故事/照片时间线，作为后续增强能力。

不建议第一阶段采集身份证号、详细住址等高敏感信息。

### 4.8 邀请、分享与群内长期使用

分享不是统一的一张卡片，而是按任务生成：

| 分享类型 | 使用场景 | 打开后的动作 |
|---|---|---|
| 家谱邀请卡 | 邀请家人加入整个家谱 | 以全谱模式预览并申请加入 |
| 成员视角卡 | 从某位成员讲解或分享家庭关系 | 直接以指定成员为视角打开家谱 |
| 补资料卡 | 请特定家人补某位成员资料 | 打开指定成员编辑任务 |
| 家谱浏览卡 | 只分享查看，不开放编辑 | 以全谱模式只读访问 |

分享卡片必须携带短期有效的邀请 token，而不是直接暴露 familyId/memberId。邀请可撤销、可设置角色、有效期和使用次数。

微信能力边界：发送分享必须由用户主动触发，产品不能自动向好友或群发送；小程序也不能承诺自动置顶或永久占据微信群入口。“常驻家庭群”的可落地实现是：

- 首次建谱后引导发起人把家谱卡片发到固定家庭群。
- 卡片标题长期稳定，例如“李家家谱｜12 位家人共同维护”。
- 每次重要更新可生成新的群分享卡片，但不做骚扰式诱导分享。
- 引导核心成员添加到“我的小程序”，方便从微信下拉入口返回。
- 记录来源群的访问场景仅用于优化回访，不把微信群成员默认视为家谱成员。

### 4.9 家庭协作与权限

用户侧只呈现三个易懂角色：

| 角色 | 权限 |
|---|---|
| 家谱管理员 | 管理成员、关系、邀请、审核、角色和家谱设置 |
| 家庭成员 | 查看全谱、切换任意成员视角；补充普通资料；提交关键关系纠错 |
| 访客 | 查看允许公开的全谱内容并切换视角，但不能修改 |

创建者默认是首位管理员，但可以增加管理员或转让创建者身份，避免家谱因单一账号停用而失管。

关键变更采用不同策略：

- 头像、简介等低风险资料：家庭成员可直接补充，保留日志。
- 姓名、父母、伴侣、合并成员等高影响信息：管理员审核。
- 管理员自己的高影响操作：直接生效，但支持历史版本回滚。

### 4.10 家庭页

家庭页承担协作，而不是重复展示成员列表：

- 家庭进度：谱中人数、共同维护人数、资料完整度。
- 待办：加入申请、关系修改、重复成员、待补资料。
- 家庭动态：谁添加或更新了谁。
- 家庭成员：参与维护的微信用户及其角色。
- 邀请家人：生成不同类型的邀请任务。

### 4.11 我的

- 最近查看：我最近打开的家谱、全谱位置和成员视角，可一键继续。
- 我的家谱：切换、退出、移交管理权。
- 我的消息：审核结果、邀请和订阅提醒。
- 隐私设置：资料可见范围、授权记录、注销账号。
- 数据管理：导出家谱、下载照片、申请删除数据。
- 帮助与反馈：图文教程、联系客服、问题反馈。

### 4.12 视觉与易用性

整体采用“现代东方家庭图谱”风格：温暖但不陈旧，庄重但不沉重，关系复杂但操作简单。详细规范见[《有谱设计风格规范》](./有谱设计风格规范.md)。

品牌关键词：温暖、可信、清晰、有生命力。

- 主色建议使用墨绿或深青，搭配暖米白背景和少量朱砂红强调，避免当前页面大面积棕色带来的传统工具感。
- 图谱连接线和节点层级保持高对比度，不用颜色单独表达性别。
- 默认正文不低于 16px，关键按钮不低于 44px 点击高度。
- 图标必须配文字，避免年长用户猜测含义。
- 表单一屏只解决一个任务；错误提示说明如何修正。
- 支持系统字体放大、弱网重试、骨架屏和明确的保存状态。

## 5. 微信云开发运营后台方案

### 5.1 后台定位

运营人员不应直接使用云数据库控制台修改业务数据。建议在微信云开发上部署独立的 Web 管理后台，供运营、客服、内容安全和产品人员使用；云开发控制台仅供研发和运维查看资源、日志、告警与部署状态。

后台遵循“默认看统计，按工单看详情”的原则。运营人员不应无理由浏览真实家谱内容，所有敏感访问都要记录操作者、原因、时间和数据范围。

### 5.2 后台角色

| 后台角色 | 主要权限 |
|---|---|
| 超级管理员 | 后台账号、角色、全局配置，人数严格限制 |
| 运营 | 数据看板、活动配置、帮助内容，不可查看敏感成员资料 |
| 客服 | 根据用户工单查询必要信息，不能直接修改家谱关系 |
| 内容安全 | 处理举报、违规图片和文本，受控查看被举报内容 |
| 技术运维 | 云函数、存储、任务、日志和告警，不处理业务内容 |
| 审计人员 | 只读查看后台操作和敏感访问日志 |

### 5.3 后台功能模块

#### A. 数据看板

- 新增用户、创建家谱、受邀查看和加入人数。
- 有效家庭数、家庭活跃率、30 日留存。
- 平均家谱成员数、关系数、资料完整度。
- 分享发起、邀请打开、全谱/成员视角查看、申请加入、首次补充漏斗。
- 云函数错误率、响应时间、存储用量和成本趋势。

#### B. 用户管理

- 根据内部 userId、工单号等检索用户。
- 查看账号状态、家谱数量、违规状态和授权记录。
- 冻结/解冻、注销进度、数据导出与删除申请。
- 不在列表页展示完整 openid、生日等敏感字段。

#### C. 家谱空间管理

- 查看家谱状态、成员规模、管理员数量、最近活跃时间。
- 处理无管理员家谱、异常大规模家谱和删除恢复。
- 家谱内容默认脱敏；只有关联有效工单并二次确认后才可临时查看。
- 运营不得直接修改关系；必要时生成修复任务交给家庭管理员确认。

#### D. 审核与举报

- 加入争议、权限争议、错误关系、违规头像/文字举报。
- 工单分配、处理时限、证据记录、申诉和结果通知。
- 支持隐藏违规内容、限制账号、恢复内容。

#### E. 内容与配置

- 新手引导、帮助中心、公告和协议版本。
- 分享卡片标题/图片模板，但不得动态注入敏感成员资料。
- 关系称谓字典、资料字段开关、灰度功能配置。
- 订阅消息模板和发送记录。

#### F. 系统与审计

- 管理员账号、角色和权限。
- 敏感数据访问日志、后台操作日志、导出日志。
- 云函数错误、定时任务、消息任务、存储异常告警。
- 数据备份、恢复演练和异常操作回滚。

## 6. 业务与数据设计

### 6.1 核心数据模型

现项目把 `fatherId`、`motherId`、`spouseId` 直接保存在成员记录中，适合演示但不适合多人协作和复杂谱系。重构后将“人物”和“关系”拆开：

| 集合 | 用途 | 关键字段示例 |
|---|---|---|
| users | 微信用户账户 | openid、状态、授权版本 |
| families | 家谱空间 | 名称、创建者、状态、隐私级别 |
| family_memberships | 用户与家谱的协作关系 | familyId、userId、角色、状态 |
| user_family_preferences | 用户在家谱中的查看偏好 | familyId、userId、viewMode、lastViewPersonId、布局 |
| persons | 家谱中的人物 | familyId、姓名、头像、基础资料、状态 |
| relations | 人物之间的关系边 | fromPersonId、toPersonId、类型、状态 |
| invitations | 邀请与分享凭证 | tokenHash、目标、角色、有效期、次数 |
| change_requests | 高影响修改申请 | 修改前后、提交人、审核状态 |
| audit_logs | 业务变更历史 | 操作者、对象、动作、快照 |
| media_assets | 家庭照片与文件 | fileId、归属、状态、审核信息 |
| reports | 举报和客服工单 | 对象、原因、处理状态 |
| notifications | 站内/订阅消息任务 | 类型、接收人、状态、发送时间 |

`relations` 使用规范化的关系边存储。第一阶段支持父母—子女、伴侣关系；兄弟姐妹等展示关系由共同父母推导，减少互相矛盾的重复字段。

### 6.2 权限校验原则

每个云函数都必须从微信上下文获取 openid，再完成以下校验：

1. 当前用户是否属于目标家谱。
2. 当前角色是否允许执行该动作。
3. 被操作人物是否在同一家谱。
4. 邀请 token 是否有效、未撤销、未超次数。
5. 高影响变更是否需要审核。

前端传入的 familyId、personId 和角色都不能作为权限依据。删除操作采用软删除，支持短期恢复；照片删除和关系清理由异步任务完成。

### 6.3 云函数/服务拆分

将当前单个 `familyFunctions` 按领域拆分，或至少在代码内模块化：

- auth：登录、账户、授权和注销。
- family：创建、设置、切换、归档和管理员交接。
- membership：加入、退出、角色和成员管理。
- person：人物资料、搜索、视角定位和合并。
- relation：添加、校验、修改、称谓计算和家谱图查询。
- invitation：创建邀请、解析 token、撤销和统计。
- workflow：审核、待办和历史回滚。
- media：上传凭证、审核、引用和清理。
- notification：站内消息和订阅消息。
- admin：仅后台服务身份可调用的运营接口。

公共层统一处理身份认证、权限、参数校验、错误码、幂等、审计和限流。

## 7. 隐私、安全与合规

- 家谱默认私密，未加入的用户只能看到邀请落地页允许展示的信息。
- 邀请链接使用随机 token，服务端保存哈希，并支持撤销和过期。
- 出生日期等资料支持“仅自己/直系亲属/全家可见”范围。
- 未成年人资料默认更严格，不在分享卡片中显示生日和照片。
- 用户上传内容需要举报入口和必要的内容安全检查。
- 用户可以退出家谱、清除查看偏好、导出有权访问的数据和申请注销。
- 数据删除需处理人物资料、关系、邀请、缓存和云存储文件。
- 后台采用最小权限和敏感操作二次确认，所有查看与导出可审计。
- 发布前补齐隐私政策、用户协议、儿童信息规则和第三方 SDK 清单。

## 8. 核心指标

### 8.1 北极星指标

**月度有效家庭数**：过去 30 天内，至少有 2 位真实家庭协作者访问，且发生过一次有效维护行为的家谱数量。

“有效维护”包括新增人物、完善资料、确认关系或审核修改；单纯打开和切换视角不算。

### 8.2 关键漏斗

1. 打开小程序 → 开始创建。
2. 开始创建 → 生成首张家谱图。
3. 生成家谱 → 发起首次邀请。
4. 邀请打开 → 浏览全谱或指定成员视角。
5. 有效浏览 → 申请加入家谱。
6. 加入 → 首次补充资料或关系。
7. 首次贡献 → 7/30 日再次访问。

### 8.3 质量指标

- 首次建谱完成时间。
- 新家谱 24 小时内录入 3 人的比例。
- 新家谱 7 日内第二位协作者加入率。
- 邀请打开率、全谱有效浏览率、成员视角使用率、加入率、首次贡献率。
- 家谱平均共同维护人数、全谱/视角模式使用分布和资料完整度。
- 关系冲突率、误删恢复率、举报处理时长。
- 云函数成功率、P95 响应时间、图片上传成功率。

## 9. 版本规划

### 9.1 MVP：建立协作闭环

必须交付：

- 新品牌与三栏信息架构。
- 从“我”开始的图形化建谱。
- 基础家谱画布、成员卡片和关系添加。
- 家谱切换。
- 全谱浏览、任意成员视角及两种模式间的无损切换。
- 邀请 token、微信分享卡片、全谱/成员视角深链和加入申请。
- 管理员/家庭成员/访客权限。
- 修改记录、软删除和基础审核。
- 运营后台的数据看板、用户/家谱状态、举报工单和审计日志。
- 隐私政策、注销和数据删除流程。

MVP 暂不追求复杂关系和精美家族故事，优先验证“创建—分享—家人加入—共同维护”。

### 9.2 V1.1：提升完整度与留存

- 待补资料任务和定向分享。
- 重复人物检测、合并和关系冲突修复。
- 家庭动态和资料完整度。
- 用户主动订阅后的生日/待办提醒。
- 家谱图片/PDF 导出和基础备份。
- 后台内容配置、灰度发布和运营漏斗。

### 9.3 V2：家庭记忆空间

- 人物故事、老照片、音频口述和时间线。
- 多种家谱布局、分支折叠和超大家谱性能优化。
- 纪念日、家庭大事记。
- GEDCOM 等标准格式的导入导出可行性评估。
- 经验证后再考虑付费存储、高清导出或实体谱册服务。

## 10. MVP 验收标准

1. 新用户不阅读帮助也能在 3 分钟内，以任意起始成员建立至少包含 3 人的家谱。
2. 用户从普通入口打开家谱时无需选择成员，直接以全谱模式看到完整关系结构。
3. 创建者可分享全谱卡片，也可从任意成员生成视角卡片；接收者分别进入对应查看模式。
4. 两位用户对同一家谱的可见内容和编辑权限符合角色规则。
5. 用户可以在两步内把任意成员设为当前视角，也能一步返回全谱，页面正确更新布局和称谓。
6. 切换查看模式或成员视角不会改变用户的查看、编辑和管理权限。
7. 添加父母、伴侣、子女时自动生成一致的双向语义关系。
8. 循环祖先、跨家谱关联和无权限修改在服务端被拒绝。
9. 高影响修改可审核、可追溯，删除人物可在设定期限内恢复。
10. 邀请过期、撤销、超次数后不能继续加入。
11. 运营人员的敏感数据访问全部有审计记录。
12. 用户可申请导出和删除有权处理的数据。
13. 弱网、重复点击和云函数重试不会创建重复人物或重复关系。

## 11. 现有项目迁移建议

现有代码可复用微信小程序工程、云开发环境、照片上传和基础 CRUD 逻辑，但产品结构和数据权限需要重做。

| 当前实现 | 重构方向 |
|---|---|
| 首页 + 家谱树 + 成员列表四栏并列 | 家谱、家庭、我的三栏，默认进入家谱 |
| 表单式选择父母/配偶 | 在人物节点上直接添加关系 |
| 本地缓存 currentFamily | 云端记录最近访问家谱并支持切换 |
| fatherId/motherId/spouseId 字段 | persons + relations 关系图模型 |
| 只有 creatorOpenid | family_memberships + 角色权限 |
| 直接传 familyId/memberId 操作 | 服务端身份、归属和权限校验 |
| 单个 familyFunctions 文件 | 领域模块 + 公共鉴权与审计层 |
| 删除后物理移除 | 软删除、关系清理、恢复机制 |
| 分享能力缺失 | 邀请 token、全谱/成员视角卡片、加入与共同补全流程 |
| 家谱树只展示有限代数 | 可递归、可折叠的关系图画布 |
| README 仍为 quickstart | 补齐产品、数据、部署和运营文档 |

建议不要在旧数据结构上持续叠加邀请和权限逻辑。先设计新集合与权限层，提供一次性迁移脚本把现有 `families`、`members` 转成新模型；迁移期间保留旧集合只读备份，验证完成后再切换。

## 12. 产品决策总结

「有谱」的核心不是把纸质家谱搬到手机，而是降低一家人共同建立和维护家谱的协作成本。产品成败取决于三个体验：一个人第一次建谱是否足够快、家人能否直接看懂全谱并在需要时切换人物视角、多人修改是否可信可控。

因此，开发优先级应固定为：**图形化全谱 > 任意成员视角 > 分享与加入 > 权限与审核 > 长期内容沉淀**。只要前四项形成闭环，“常驻家庭群、供家人长期使用”才会从一句定位变成真实使用行为。
